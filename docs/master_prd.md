# PRD: 반도체 시료 생산주문관리 시스템

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | 반도체 시료 생산주문관리 시스템 (S-Semi) |
| 개발 환경 | C++ 콘솔 애플리케이션 |
| 데이터 저장 | JSON 파일 기반 영속성 |
| 아키텍처 | MVC 패턴 |

---

## 2. 배경 및 목적

가상의 반도체 회사 **S-Semi**는 연구소·팹리스 업체·대학 연구실 등에 반도체 시료(Sample)를 생산·납품한다.
주문량 급증으로 엑셀/메모장 기반 관리의 한계가 드러났으며, 아래 문제를 해결하기 위해 시스템을 개발한다.

- 주문 처리 여부를 한눈에 파악 불가
- 공정 예약 후 완성 시점 불명확
- 충분한 재고가 있음에도 불필요한 생산 가동

---

## 3. 시스템 아키텍처

### 3.1 전체 구조

```
콘솔 입력 → Controller → Model (비즈니스 로직)
                              ↓
                        JSON 파일 (영속성)
                              ↓
Controller ← Model → View (콘솔 출력)
```

### 3.2 생산 라인 개요

- 공장의 하나의 생산 라인은 **시료를 1개씩** 순차 생산
- **주문이 들어온 시료에 대해서만** 생산
- 생산 완료 시 주문 상태 자동 변경

---

## 4. 주문 상태 흐름

```
[주문 접수]
    │
    ▼
RESERVED ──────────────► REJECTED
    │           (거절)
    │ (승인)
    ├─── 재고 충분 ────────► CONFIRMED ──► RELEASE
    │                                   (출고+재고차감)
    └─── 재고 부족 ────────► PRODUCING ──► CONFIRMED ──► RELEASE
                           (생산 중)     (생산 완료)   (출고+재고차감)
```

| 상태 | 의미 | 모니터링 포함 |
|------|------|:---:|
| RESERVED | 주문 접수 대기 | O |
| REJECTED | 주문 거절 (비정상 흐름) | X |
| PRODUCING | 승인 완료, 재고 부족으로 생산 중 | O |
| CONFIRMED | 승인 완료, 출고 대기 | O |
| RELEASE | 출고 완료 | O |

---

## 5. 기능 요구사항

### 5.1 메인 메뉴

- 기능별 선택 화면 표시
- 전체 시료에 대한 요약 정보 표시 (재고 현황 스냅샷)
- 화면 구성은 자유롭게 설계
- **메뉴 진입 시 생산 완료 여부를 자동으로 체크** (lazy 처리)

| 번호 | 메뉴 | 설명 |
|------|------|------|
| 1 | 시료 관리 | 시료 등록 / 목록 조회 / 검색 |
| 2 | 주문 | 주문 접수 / 승인·거절 처리 |
| 3 | 모니터링 | 상태별 주문 수 및 재고 현황 |
| 4 | 출고 처리 | CONFIRMED 주문 출고 실행 |
| 5 | 생산 라인 | 현재 생산 현황 및 대기 큐 확인 |
| 0 | 종료 | 프로그램 종료 |

### 5.2 시료 관리 (Sample Management)

시스템의 **기본 단위**. 등록된 시료만 주문 가능.

#### 5.2.1 시료 등록

| 입력 필드 | 타입 | 설명 |
|-----------|------|------|
| 시료 ID | string | **사용자가 직접 입력**하는 고유 식별자 |
| 시료 이름 | string | 시료명 |
| 평균 생산시간 | int | 시료 1개 생산에 소요되는 평균 시간 **(단위: 분)** |
| 수율 | double | 정상 시료 / 총 생산 시료 (0 < 수율 ≤ 1.0) |

> **수율 예시:** 100개 생산 중 정상 90개 → 수율 = 0.9

- 동일 ID 또는 동일 이름 중복 등록 방지 처리
- 등록 성공 시 JSON 파일에 즉시 저장

#### 5.2.2 시료 조회

- 등록된 모든 시료 목록 출력
- 각 시료의 **현재 재고 수량** 함께 표시

#### 5.2.3 시료 검색

- 시료 이름(부분 일치)으로 검색
- 검색 결과: 시료 ID, 이름, 수율, 평균 생산시간, 재고 수량

---

### 5.3 시료 주문 (Order)

#### 5.3.1 시료 예약 (주문 접수)

| 입력 필드 | 타입 | 설명 |
|-----------|------|------|
| 시료 ID | string | 등록된 시료만 허용 |
| 고객명 | string | 주문 고객 이름 |
| 주문 수량 | int | 양의 정수 |

- 입력 완료 시 주문 상태: **RESERVED**
- 주문 ID는 시스템 자동 생성 (형식: `ORD-001`)
- 미등록 시료 ID 입력 시 오류 메시지 출력

---

### 5.4 주문 승인/거절

#### 5.4.1 접수된 주문 목록 조회

- **RESERVED** 상태의 주문 목록만 표시
- 표시 항목: 주문 ID, 시료명, 고객명, 수량, 접수 일시

#### 5.4.2 주문 승인

승인 시 재고 상황에 따라 **자동 분기** 처리:

```
현재 재고 >= 주문 수량
    → 주문 상태: RESERVED → CONFIRMED
    → 재고는 변경 없음 (RELEASE 출고 시점에 차감)

현재 재고 < 주문 수량
    → 부족분 계산: 부족분 = 주문 수량 - 현재 재고
    → 실 생산량 = ceil(부족분 / (수율 × 0.9))
    → 총 생산 시간 = 평균 생산시간(분) × 실 생산량
    → 생산 작업을 생산 큐(FIFO)에 등록
    → 주문 상태: RESERVED → PRODUCING
```

> **생산 수량 공식 설명:**
> - 수율(`yield`) × 0.9 는 오차를 포함한 실질 수율 (안전 계수 0.9 적용)
> - 예: 부족분=10, 수율=0.9 → ceil(10 / (0.9×0.9)) = ceil(10/0.81) = ceil(12.34) = **13개 생산**

#### 5.4.3 주문 거절

- 선택한 주문 즉시 **REJECTED** 전환
- REJECTED는 이후 모니터링에서 제외

---

### 5.5 모니터링 (Monitoring)

담당자가 시스템 전체 상태를 한눈에 파악.

#### 5.5.1 주문량 확인

- 상태별 주문 목록 및 건수 표시
- 포함 상태: **RESERVED / PRODUCING / CONFIRMED / RELEASE**
- 제외 상태: REJECTED

#### 5.5.2 재고량 확인

| 재고 상태 | 조건 | 표기 |
|-----------|------|------|
| 여유 | 현재 재고 > 유효 주문 수량 합계 | [여유] |
| 부족 | 0 < 현재 재고 ≤ 유효 주문 수량 합계 | [부족] |
| 고갈 | 현재 재고 = 0 | [고갈] |

- 각 시료별로 현재 재고 수량과 상태 표시

---

### 5.6 생산 라인 (Production Line)

#### 5.6.1 생산 현황 표기

현재 생산 중인(Active) 작업의 정보 표시:

| 표시 항목 | 내용 |
|-----------|------|
| 주문 ID | 어떤 주문에 대한 생산인지 |
| 시료명 | 생산 중인 시료 |
| 실 생산량 | 목표 생산 수량 |
| 총 생산 시간 | 예상 소요 시간 (분) |
| 생산 시작 시각 | 생산 시작 unix timestamp |
| 경과 시간 | 현재까지 경과된 시간 (분) |
| 예상 완료 시각 | 시작 시각 + 총 생산 시간 |

#### 5.6.2 생산 완료 처리 — Lazy + 프로그램 재시작 내구성

**핵심 원칙:** 생산 완료 여부는 요청 시점에 lazy하게 판단한다.

- 생산 시작 시각을 **unix timestamp(초)** 로 JSON에 저장
- **체크 트리거**: 메인 루프 반복 시, 생산 라인 메뉴 진입 시
- **완료 판단**: `현재 시각 - 시작 시각(초) >= 총 생산 시간(분) × 60`
- **프로그램이 꺼져 있던 동안**에도 시간이 흘렀으므로, 재시작 시 과거 완료분을 자동으로 일괄 처리

```
완료 처리 순서:
1. activeJob의 실 생산량 전부를 해당 시료 재고에 추가
2. 해당 주문 상태: PRODUCING → CONFIRMED
3. activeJob 제거, JSON 저장
4. 큐에 대기 항목이 있으면 다음 작업 자동 시작 (startTime = 현재 시각)
5. 새로 시작된 작업도 즉시 완료 조건 재체크 (큐에 쌓인 동안 완료 시간이 지났을 수 있음)
```

#### 5.6.3 시간 Mocking (테스트 지원)

실제 분 단위 대기 없이 빠른 검증을 위해 **ITimeProvider 인터페이스** 분리:

```
ITimeProvider (interface)
├── RealTimeProvider   → std::chrono::system_clock::now() 사용
└── MockTimeProvider   → mock 시각 수동 조작 (advance, setTime)
```

- 프로그램 시작 시 `--mock-time` 인수로 MockTimeProvider 활성화
- MockTimeProvider는 `advance(minutes)` 호출로 시각을 임의로 앞당김
- 모든 시간 연산은 반드시 ITimeProvider를 통해서만 수행

#### 5.6.4 대기 주문 확인

- 생산 큐에 대기 중인 작업 목록 출력
- **스케줄링 전략: FIFO** (선입선출)
- 표시: 대기 순번, 주문 ID, 시료명, 주문 수량, 실 생산량, 예상 소요 시간(분)

---

### 5.7 출고 처리 (Shipment)

- **CONFIRMED** 상태 주문 목록 표시
- 특정 주문 선택 후 출고 실행
- 출고 처리 시:
  1. 해당 주문 수량만큼 **재고 차감**
  2. 주문 상태: **CONFIRMED → RELEASE**

> 재고는 오직 RELEASE 출고 처리 시점에만 차감된다. CONFIRMED 전환 시에는 재고를 변경하지 않는다.

## 6. 데이터 모델

### 6.1 시료 (Sample) — `data/samples.json`

```json
{
  "samples": [
    {
      "id": "S-001",
      "name": "AlGaN 시료",
      "avgProductionTime": 5,
      "yieldRate": 0.9,
      "currentStock": 50
    }
  ]
}
```

| 필드 | 타입 | 설명 |
|------|------|------|
| id | string | 사용자 입력 고유 ID |
| name | string | 시료 이름 |
| avgProductionTime | int | 평균 생산시간 **(분)** |
| yieldRate | double | 수율 (0.0 초과 ~ 1.0 이하) |
| currentStock | int | 현재 재고 수량 |

---

### 6.2 주문 (Order) — `data/orders.json`

```json
{
  "orders": [
    {
      "id": "ORD-001",
      "sampleId": "S-001",
      "customerName": "삼성연구소",
      "quantity": 20,
      "status": "RESERVED",
      "createdAt": "2026-05-08T10:00:00"
    }
  ]
}
```

| 필드 | 타입 | 설명 |
|------|------|------|
| id | string | 자동 생성 주문 ID (ORD-NNN) |
| sampleId | string | 연결된 시료 ID |
| customerName | string | 고객명 |
| quantity | int | 주문 수량 |
| status | string | RESERVED / PRODUCING / CONFIRMED / RELEASE / REJECTED |
| createdAt | string | 주문 접수 일시 (ISO 8601) |

---

### 6.3 생산 관리 — `data/production.json`

```json
{
  "activeJob": {
    "orderId": "ORD-002",
    "sampleId": "S-001",
    "shortage": 5,
    "actualProductionQty": 7,
    "totalProductionTimeMin": 35,
    "startTimeUnix": 1746691200
  },
  "queue": [
    {
      "orderId": "ORD-003",
      "sampleId": "S-002",
      "shortage": 10,
      "actualProductionQty": 14,
      "totalProductionTimeMin": 70
    }
  ]
}
```

| 필드 | 타입 | 설명 |
|------|------|------|
| orderId | string | 연결된 주문 ID |
| sampleId | string | 생산 대상 시료 ID |
| shortage | int | 부족분 (주문 수량 - 승인 시 재고) |
| actualProductionQty | int | 실 생산량 = ceil(부족분 / (수율 × 0.9)) |
| totalProductionTimeMin | int | 총 생산 시간 **(분)** = avgProductionTime × actualProductionQty |
| startTimeUnix | int64 | 생산 시작 unix timestamp (activeJob만 존재) |

## 7. 비즈니스 규칙

| # | 규칙 |
|---|------|
| BR-01 | 등록되지 않은 시료는 주문 불가 |
| BR-02 | RESERVED 상태 주문만 승인/거절 가능 |
| BR-03 | 승인 시 재고 충분 → 즉시 CONFIRMED 전환 (이 시점에 재고 차감 없음) |
| BR-04 | 승인 시 재고 부족 → 생산 큐에 등록, PRODUCING 전환 |
| BR-05 | 부족분 = 주문 수량 - 현재 재고 수량 |
| BR-06 | 실 생산량 = ceil(부족분 / (수율 × 0.9)) |
| BR-07 | 총 생산 시간(분) = 평균 생산시간(분) × 실 생산량 |
| BR-08 | 생산 큐는 FIFO 방식으로 처리 |
| BR-09 | 생산 완료 시 실 생산량 **전부**를 해당 시료 재고에 추가 후 CONFIRMED 전환 |
| BR-10 | 출고(RELEASE) 처리 시에만 주문 수량만큼 재고 차감 |
| BR-11 | CONFIRMED 상태 주문만 출고 처리 가능 |
| BR-12 | REJECTED 주문은 모니터링 집계에서 제외 |
| BR-13 | 생산 라인은 한 번에 하나의 작업만 처리 (단일 생산 라인) |
| BR-14 | 수율은 0 초과 1.0 이하 값만 허용 |
| BR-15 | 생산 완료 여부는 lazy 판단: 현재 시각 - 시작 시각 >= 총 생산 시간(분) × 60초 |
| BR-16 | 프로그램 재시작 시 기존 activeJob 완료 여부를 unix timestamp 비교로 자동 처리 |
| BR-17 | 큐 처리 중 새로 시작된 다음 작업도 즉시 완료 여부를 재귀 체크 |
| BR-18 | 동일 시료에 대한 여러 PRODUCING 주문은 각각 독립 생산 작업으로 큐에 등록 |

> **데이터 영속성**: 앱을 재시작해도 데이터와 생산 진행 상태(시작 시각 기반)를 정확히 유지

## 10. 확정된 설계 결정 사항

| # | 항목 | 결정 내용 |
